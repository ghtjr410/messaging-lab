# 이벤트 기반 시스템 설계 원칙 — 6-Layer Framework

---

## 이 문서는 무엇인가

Step 0~7을 관통하는 설계 원칙을 계층으로 정리한 것이다. 각 레이어는 아래 레이어 위에서만 의미가 있다. 아래가 무너지면 위는 성립하지 않는다.

이 프레임워크의 용도는 두 가지다.

1. **학습 완료 후 복습** — Step 0~7에서 경험한 것을 원칙으로 체계화한다.
2. **실무 설계 검증** — "이 설계가 Layer 몇에서 탈락하는가?"로 즉시 판단한다.

---

## Layer 0: 내구성 (Durability)

본질 키워드: **"기록"**

비즈니스 사건이 발생했다는 사실을 내구성 있는 저장소에 기록한다. 메모리는 휘발된다. 프로세스는 죽는다. 유일하게 신뢰할 수 있는 건 커밋된 DB다.

**이 레이어가 판별하는 것:** "이 사건의 기록이 프로세스가 죽어도 살아남는가?"

```
@Async 이벤트 발행          → 메모리에만 존재 → 탈락
@TransactionalEventListener → 메모리에만 존재 → 탈락
Outbox INSERT (같은 TX)     → DB에 커밋됨    → 통과
```

**Step 매핑:** Step 2에서 `@Async`의 메모리 휘발 한계를 체험하고, Step 3에서 Event Store로 해결했다.

---

## Layer 1: 원자성 (Atomicity)

본질 키워드: **"묶음"**

비즈니스 상태 변경과 사건 기록은 하나의 트랜잭션으로 묶는다. 둘 중 하나만 성공하면 시스템이 모순 상태에 빠진다. "주문은 생성됐는데 이벤트 기록은 없다"가 발생하면 안 된다.

**이 레이어가 판별하는 것:** "비즈니스 로직과 이벤트 기록 사이에 네트워크 홉이나 별도 트랜잭션이 존재하는가?"

```java
// 모순 가능 — 별도 트랜잭션
orderRepository.save(order);        // TX1 커밋
kafkaTemplate.send("order", event); // 네트워크 홉 — 여기서 실패하면 모순

// 모순 불가 — 같은 트랜잭션
@Transactional
void createOrder() {
    orderRepository.save(order);          // 같은 TX
    outboxRepository.save(outboxMessage); // 같은 TX → 둘 다 성공하거나 둘 다 실패
}
```

Layer 0이 "기록해라"고 했고, Layer 1이 **"비즈니스랑 묶어서 기록해라"**로 구체화한 것이다.

**Step 매핑:** Step 3에서 "도메인 저장 + 이벤트 기록 = 같은 TX"로 원자성을 확보했고, Step 6에서 "TX 안에서 Kafka 발행하면 안 되는 이유"로 이 원칙을 재확인했다.

---

## Layer 2: 전달 보장 (Delivery Guarantee)

본질 키워드: **"최소 1회"**

기록된 사건은 반드시 수신 측에 최소 1회 전달한다. 기록이 DB에 있으니까 재시도가 가능하다. 실패하면 다시 읽어서 다시 보내면 된다.

**이 레이어가 판별하는 것:** "발행에 실패했을 때, 다시 시도할 수 있는 근거가 존재하는가?"

```
메모리 기반 이벤트 → 실패하면 근거 없음 → 재시도 불가
Outbox 기반       → PENDING 상태로 DB에 남아있음 → 재시도 가능
```

릴레이 전략이 여기서 결정된다. Polling은 주기적으로 PENDING을 SELECT해서 발행하고, CDC(Debezium)는 binlog를 실시간으로 읽어서 발행한다. 방식은 달라도 본질은 같다 — **"DB에 기록된 것을 읽어서 브로커로 옮긴다."**

재시도 정책도 여기에 속한다. 지수 백오프, 최대 N회, 초과 시 FAILED + 알림. 무한 재시도는 하지 않되, 기록은 남기니까 사람이 개입해서 복구할 수 있다.

**Step 매핑:** Step 3의 스케줄러가 PENDING을 조회해서 처리하는 것, Step 6의 릴레이가 Kafka로 발행하고 실패 시 PENDING을 유지하는 것이 이 레이어다.

---

## Layer 3: 확인 (Acknowledgement)

본질 키워드: **"응답"**

전달했다는 사실을 상대방의 응답으로 확인한다. "보냈다"와 "도착했다"는 다르다.

**이 레이어가 판별하는 것:** "상대방이 받았다는 응답을 어떤 수준으로 확인하는가?"

```
acks=0   → 확인 안 함      → 유실 가능
acks=1   → 리더만 확인     → 리더 장애 시 유실 가능
acks=all → ISR 전체 확인   → 유실 사실상 불가
```

`acks=all` ACK를 받으면 PENDING → SENT로 상태를 바꾼다. 이 시점에서 SENT는 "브로커에 내구적으로 적재됨"을 의미한다.

Layer 2와 Layer 3은 실제 코드에서는 한 동작 안에서 일어난다 — 릴레이가 `kafka.send()`하고 ACK를 받고 SENT로 바꾸는 것이 Layer 2(재시도 가능)와 Layer 3(적재 확인)을 동시에 수행하는 것이다. 개념적으로 분리하는 이유는, "재시도할 수 있는가"(Layer 2)와 "도착을 어떤 수준으로 확인하는가"(Layer 3)가 **독립적으로 설정 가능한 별도 관심사**이기 때문이다. acks 수준을 바꿔도 재시도 로직은 그대로이고, 재시도 정책을 바꿔도 acks 수준은 그대로다.

**Step 매핑:** Step 7의 "acks=all이어도 중복이 생기는 이유" 섹션에서 다뤘다.

## Layer 4: 멱등성 (Idempotency)

본질 키워드: **"같은 결과"**

같은 메시지가 2번 와도 결과는 1번과 동일해야 한다. At-least-once 전달은 구조적으로 중복을 만든다. Layer 2에서 "최소 1회"를 보장한 대가다.

**이 레이어가 판별하는 것:** "같은 이벤트가 N번 처리되어도 비즈니스 상태가 올바른가?"

```java
@Transactional
public void handleOrderCreated(OrderCreatedEvent event) {
    if (processedEventRepository.existsByEventId(event.getEventId())) {
        return; // 이미 처리됨 → 스킵
    }
    pointService.record(event);
    processedEventRepository.save(event.getEventId());
}
```

도메인 상태가 이미 멱등한지 먼저 확인하고(상태 머신), 아니면 event_handled / upsert / version 비교를 추가한다. 이게 Consumer 멱등이고, 전체 파이프라인의 **최종 방어선**이다.

**Step 매핑:** Step 7의 멱등 패턴(상태 머신, event_handled, upsert, version 비교)과 DLQ가 이 레이어다.

---

## Layer 5: 관측 (Observability)

본질 키워드: **"가시성"**

시스템의 모든 상태 전이를 사람이 볼 수 있어야 한다. 자동화의 끝에는 반드시 사람이 있다. 시스템이 스스로 복구할 수 없는 상태에 도달했을 때, 사람이 상황을 파악하고 개입할 수 있어야 한다.

**이 레이어가 판별하는 것:** "지금 이 순간 FAILED 상태인 메시지가 몇 개인지, 왜 실패했는지, 언제부터 밀려 있는지 알 수 있는가?"

FAILED 메시지 수, 평균 발행 지연, 재시도 횟수 분포, Consumer lag. 이런 것들이 대시보드에 보여야 한다. 안 보이면 장애가 아니라 **미스터리**가 된다.

**Step 매핑:** Step 7의 "DLQ에 쌓인 걸 누가 보는가", "발행 실패는 유실이 아니라 미발행 상태", "FAILED + 알림" 등이 이 레이어의 시작점이다. messaging-lab에서 깊이 다루지는 않았지만, 모든 Step에서 상태 전이(PENDING → SENT → PROCESSED / FAILED)를 설계한 것이 관측의 기반이다.

---

## 전체 구조

```
Layer 5: 관측       — "가시성"   — 사람이 개입할 수 있는 근거
Layer 4: 멱등성     — "같은 결과" — 중복이 와도 비즈니스가 깨지지 않는 방어
Layer 3: 확인       — "응답"     — 도착했다는 상대방의 보장
Layer 2: 전달 보장  — "최소 1회" — 기록이 있으니 재시도 가능
Layer 1: 원자성     — "묶음"     — 비즈니스와 기록의 일체화
Layer 0: 내구성     — "기록"     — 모든 것의 시작점
```

**아래 레이어가 무너지면 위 레이어는 의미가 없다.** 기록이 없으면 묶음이 의미 없고, 묶음이 없으면 전달 보장이 모순되고, 전달 보장이 없으면 멱등성만으로는 유실을 막을 수 없다.

| "모니터링은 나중에 붙이면 되죠?" | Layer 5 — 장애를 볼 수 없다 |
